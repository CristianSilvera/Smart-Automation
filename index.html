Smart Automation: Functional Testing with AI

This document is an excerpt of the main idea presented in the final project of the defense for the 2024 Diploma in Quality Management and Software Testing, delivered by the CES (Software Testing Center) of the Permanent Education Program at the Faculty of Engineering, UDELAR.

The original work from which this excerpt is derived was authored by Valentina Prado and Cristian Silvera.

It explains the concepts, processes, and benefits of integrating Artificial Intelligence into functional testing through an intelligent automation system with self-healing capabilities.

To facilitate understanding, I made a simplification of the work carried out. This abstraction aims to explain the lifecycle of the three fundamental components involved in the system.

System Proposal

The core idea of this work is to design a self-healing automation system, composed of three main elements:

  1. Automated Test:
    • Executes tests on an application (e.g., an Android calculator).
    • Monitors the application logs for errors that may occur due to changes in selectors or the interface.
  1. LLM-RAG Model (Language Model + Retrieval-Augmented Generation):
    • Receives queries derived from logs with error details.
    • Processes and analyzes the data through vectorization and a query engine, generating an enriched response, such as updated and functional selectors.
  1. Application:
    • Acts as the environment under test.
    • Provides the data (logs) that trigger feedback in the system.

System Flow:

  1. The Automated Test runs a script and monitors the application logs.
  1. When an error occurs, the Test generates a query with the problem details and sends it to the LLM-RAG Model.
  1. The Model processes the query, searches its vectorized database, and returns a response with updated information (new selectors).
  1. The Test applies the information to correct the error and re-execute the test (Self-Healing).
  1. If the correction fails after several attempts, the system logs the error details for manual intervention.

Purpose:

The goal of the system is to reduce the reliance on human intervention in test automation, allowing tests to automatically adapt to changes in the application, thereby improving efficiency and reliability in the testing process.

Definitions

Definition of Self-Healing

Self-Healing in the context of test automation refers to the ability of an automated test to automatically recover from temporary or intermittent failures without manual intervention. This includes the ability to handle situations where user interface (UI) elements may not be immediately available due to synchronization issues, loading problems, or changes in the UI structure.

Characteristics of a Self-Healing Test Approach

To implement a Self-Healing approach in your automated tests, they should include the following characteristics:

  1. Multiple Localization Strategies:
    • Use multiple selectors to locate an element. If one selector fails, try another.
    • Example: Use accessibility id, resource id, xpath, css selector, etc.
  1. Automatic Retries:
    • Implement retry logic to try finding and interacting with elements several times before failing.
    • Example: Retry up to 3 times with a wait interval between attempts.
  1. Exception Handling:
    • Capture and handle exceptions so the test can continue even if a step fails temporarily.
    • Example: If an element is not found, capture the exception and retry.
  1. Custom Wait Conditions:
    • Define custom wait conditions to handle specific application situations.
    • Example: Wait for an element to disappear before continuing.
  1. Screenshot on Failures:
    • Take screenshots in case of failures to facilitate diagnosis.
    • Example: Save a screenshot every time a step fails.
  1. Detailed Logs:
    • Add detailed logs to better understand what is happening during test execution.
    • Example: Log each retry attempt and any exceptions captured.
  1. Application State Verification:
    • Verify the application's state before performing an action to ensure it is in the expected state.
    • Example: Verify that certain elements are visible or that certain text is present.
  1. Use of Environment Variables:
    • Use environment variables to configure parameters like the number of retries and wait time.
    • Example: Configure the number of retries and wait time through environment variables.

Definition of LLM-RAG

  1. LLM:
    • Large Language Model: Refers to large language models designed to understand and generate natural language text. These models are trained with large amounts of textual data and can perform a variety of tasks such as answering questions, translating text, summarizing documents, and more.
  1. RAG:
    • Retrieval-Augmented Generation: A technique that combines information retrieval with text generation. In this approach, a language model uses information retrieved from a database or a document index to improve the quality and accuracy of the generated responses. This is useful when specific and up-to-date information is needed that may not necessarily be included in the training data of the language model.

Diagram of a RAG:

1. RAG (Retrieval-Augmented Generation):

This component focuses on retrieving relevant information from a massive dataset and contextualizing it before responding to the query.

  1. Gigantic Dataset:
    • A massive collection of data that may consist of websites, wikis, books, or other sources. This data is processed and transformed into useful representations.
  1. Vectorized Data Chunks:
    • The original data is divided into small chunks and converted into vectors (vectorization is the process of transforming textual data into a numerical representation that machine learning algorithms can process). This enables efficient search and correlation based on similarities.
  1. Base LLM (Large Language Model):
    • The language model (LLM) serves as the foundation for interpreting the user's initial query. It generates an enriched query that combines context and relevance.
  1. Query Engine:
    • This component searches the vectorized dataset using the query generated by the LLM. Its goal is to retrieve the most relevant data chunks for the initial question.
  1. Retrieved Dataset:
    • The retrieved documents or data are combined and provided as additional context for the LLM to generate a more precise response.

2. ACTION:

This part of the model focuses on acting on the generated information to execute specific tasks.

  1. Response:
    • The LLM uses the context provided by the retrieved documents to generate an accurate response or action plan.
  1. Task Execution:
    • At this stage, the model's response is not only textual but may include executing specific tasks (e.g., invoking APIs, performing calculations, running scripts, etc.).
  1. Feedback:
    • A feedback loop is generated where the results of the execution are evaluated, and, if necessary, the initial response is adjusted to improve the outcome.
  1. Output:
    • Finally, a validated and optimized result is delivered as the final output to the user.

Purpose and Advantages of the Model:

  • Combines retrieval and generation: The model leverages massive external data to enhance the accuracy and contextualization of the LLM-generated responses.
  • Execution capability: It goes beyond answering questions, enabling task execution based on instructions.
  • Adaptability: Continuously learns and improves through the feedback loop.

This triggers the information update script used by the model to provide a response, which communicates with the test and provides the necessary components for its self-healing.

Diagram Integration Explanation:

Explanation of Component Placement in the Diagram:

  1. Application:
    • Location: Positioned before "Task Execution" in the ACTION section.
    • Role: Serves as the system under test (e.g., an Android calculator), generating logs (feedback) for the model and executing actions tested by the automated test.
    • Connections:
      • Receives commands from "Task Execution."
      • Generates logs used to identify problems via the "Base LLM."
  1. Automated Test:
    • Location: Placed between "Task Execution" and the Application.
    • Role: Executes tests on the application, monitors logs for errors, and communicates queries to the RAG model.
    • Connections:
      • Receives commands from "Task Execution."
      • In case of errors, sends logs to the "Query Engine" of the RAG model for analysis.
  1. Selector Mapping Script:
    • Location: Located within the "Query Engine" of the RAG model.
    • Role: Processes queries and generates an updated list of selectors to address issues detected in the application.
    • Connections:
      • Takes input from the "Base LLM."
      • Outputs updated selectors to "Task Execution."

Integrated Flow:

  1. The Automated Test executes scripts on the Application and monitors logs.
  1. If an error is detected, the logs are converted into a query for the Base LLM.
  1. The Query Engine processes the query and runs the Selector Mapping Script to generate an updated list of selectors.
  1. The new selectors are sent to Task Execution, which uses them to adjust the Automated Test.
  1. The Automated Test re-executes the tests with the updated selectors.
  1. If the tests succeed, no further queries are made to the model. If errors persist, the flow repeats up to a predefined limit, after which the issue is logged for manual intervention.

Summary of the Interaction

Based on the proposed idea, I have designed a conceptual framework to integrate a flow with Self-Healing Testing powered by an LLM-RAG model. This flow connects three main components: the automated test, the application (app), and the LLM-RAG model. The process includes error detection, automated queries to the model, and self-correction via selector synchronization.

System Overview

Main Components

  1. Automated Test:
    • Executes scripts to validate the application's functionality.
    • Monitors logs in real-time.
    • Triggers a query to the LLM-RAG model when an error is detected.
  1. Application (e.g., Android Calculator App):
    • The system under test, containing dynamic selectors that may break during system changes.
  1. LLM-RAG Model (Adapted):
    • Processes enriched queries.
    • Generates updated selectors, prioritized by functionality.
    • Integrates a vectorized data search engine (Query Engine).
    • Returns a response to the automated test for executing self-healing.
  1. Component Update and Synchronization Script:
    • Maps affected components based on log information (errors trigger the update and synchronization actions).
    • Stores the mapped information in the model for processing and query preparation.

Detailed Flow

1. Initial Execution of the Automated Test

  • The automated test interacts with the application using known selectors.
  • Test logs are monitored to detect errors.

2. Error Detection and Query Trigger

  • If the test fails due to selector-related issues (e.g., missing or incorrect selector):
    • The logs are analyzed automatically.
    • A structured query is generated with error details (e.g., error type, broken selector, functional context).
    • The application is mapped, updating and synchronizing information in the model.

3. Query Sent to the LLM-RAG Model

  • The query is sent to the LLM-RAG model, which follows these steps:
    • Vectorized Database: Data about the application and its selectors is stored as vectorized chunks for efficient searches (enabled by the update and synchronization script that sends the data).
    • Base LLM: Processes the query and enriches it with additional context.
    • Query Engine: Searches the vectorized data for relevant and updated selectors.
    • Generated Response: Returns a prioritized list of corrected or alternative selectors based on functionality (details about the control structure traversing this data are provided in the annex).

4. Test Self-Healing

  • The automated test receives the model's response dynamically during execution using self-healing techniques, which update selectors in real-time.
  • The test automatically retries using the updated selectors.

5. Log Monitoring (Feedback Loop)

  • If the test is successful:
    • The flow stops, and execution proceeds as normal.
    • Metrics are stored in the execution history for model feedback (creating a feedback loop where results are evaluated and the initial response is adjusted if necessary to improve outcomes).
  • If the test fails again:
    • A new query is generated for the model, with a maximum of X attempts.
    • If unresolved after multiple attempts, the error details are stored to facilitate manual debugging.
    • Feedback is provided to the model.

Example of Self-Healing Implementation:

The following code corresponds to mobile automation for the Calculator on the Android operating system, where Self-Healing is effectively implemented.

Here is a detailed analysis of how it was implemented:

1. Multiple Localization Strategies

The method getElementWithBackupSelectors tries to locate an element using multiple selectors. If one selector fails, it attempts with the next. This is a form of Self-Healing as it allows the test to continue even if the first selector does not work.

private async getElementWithBackupSelectors(selectors: string[]) {
    let lastError: Error | null = null;
    for (let selector of selectors) {
        try {
            const element = await $(selector); // Wait for the element
            if (await element.isDisplayed()) {
                return element;
            }
        } catch (error) {
            lastError = error; // Store the last error for reporting
        }
    }
    // If none of the selectors work, throw the last encountered error
    if (lastError) throw lastError;
    throw new Error('Could not locate the element with any of the selectors.');
}

2. Automatic Retries

The retry function implements retry logic with exponential backoff. This means that if an attempt fails, it waits before retrying, with the wait time increasing exponentially with each retry. This is useful for handling situations where elements may not be immediately available due to synchronization or loading issues.

private async retry(fn, retries = 3, delay = 1000) {
    let attempt = 0;
    while (attempt < retries) {
        try {
            return await fn();
        } catch (error) {
            if (attempt === retries - 1) {
                throw error;
            }
            const nextDelay = delay * Math.pow(2, attempt); // Exponential backoff
            console.log(`Attempt ${attempt + 1} failed. Retrying in ${nextDelay} ms...`);
            await new Promise(resolve => setTimeout(resolve, nextDelay)); // Wait with backoff
            attempt++;
        }
    }
}

3. Exception Handling

The tryClickButton method tries to click a button and handles exceptions. If the button is not found, it attempts to locate an alternative selector. This is another form of Self-Healing as it allows the test to proceed even if the first click attempt fails.

private async tryClickButton(btn) {
    try {
        // Wait until the button is visible
        await btn.waitForDisplayed({ timeout: 5000 });
        await btn.click(); // Perform the click
    } catch (error) {
        // If the button is not found, try locating an alternative selector
        console.log(`Failed to find the button with the initial locator. Trying an alternative.`);
        await this.retry(() => btn.waitForDisplayed({ timeout: 5000 }));
        await btn.click();
    }
}

4. Using Multiple Strategies for Buttons

The buttons btnNumero, btnMas, and btnIgual use multiple localization strategies, which enhances the robustness of the test.

get btnNumero() {
    return (numeroDeseado: string) => this.getElementWithBackupSelectors([
        `~${numeroDeseado}`, // Use ~ correctly without string templates
        `android=new UiSelector().description("${numeroDeseado}")`, // Use description as a backup
    ]);
}

get btnMas() {
    return this.getElementWithBackupSelectors([
        '~máss',
        'android=new UiSelector().description("sumar")',
        'android=new UiSelector().resourceId("com.google.android.calculator:id/btn_plus")',
        '~más'
    ]);
}

get btnIgual() {
    return this.getElementWithBackupSelectors([
        '~igual a',
        'android=new UiSelector().description("igual")',
        'android=new UiSelector().resourceId("com.google.android.calculator:id/btn_eq")',
    ]);
}

Note: While the code implements Self-Healing effectively through the use of multiple localization strategies, automatic retries with exponential backoff, and exception handling, it is an example implemented locally and does not integrate logic connecting to the LLM-RAG model.

Conclusion

The use of Artificial Intelligence and Self-Healing techniques in functional test automation optimizes resources, improves resilience, and reduces dependence on manual updates. This approach adapts to changes in applications and enhances the quality of tests while ensuring traceability through the historical logging of information. In an agile environment, these tools represent a crucial step towards more efficient and intelligent testing systems.
